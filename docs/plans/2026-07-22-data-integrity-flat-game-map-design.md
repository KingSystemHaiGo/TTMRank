# 数据完整性与扁平游戏地图设计

## 目标

这次升级同时解决两个直接影响判断的问题：TapTap iOS 排行榜采集不完整，以及“游戏宇宙”使用了与全站不一致的拟真空间表达。原始排行榜继续完整保留，现有游戏分析继续作为主要分析入口；新地图只服务于快速浏览 TapTap制造游戏，不替代精确榜位与分析表格。

完成标准如下：

- 每个平台、每张榜单的唯一游戏数必须等于 TapTap 当前响应中的 `total`，否则整轮采集失败并保留上一版 Pages。
- 原始榜位来自源分页位置，不能因某页少于常规页长而压缩名次。
- 非新非热榜只收录进入过其他榜单、且未进入 Android 或 iOS 任何热门榜、新品榜的游戏。
- 游戏地图不使用透视、球体、核心/外圈、自转或拖动；每个可见标记都对应一款真实游戏和可读数据。
- 首页不加载 Three.js 或 PixiJS；受限设备和省流量模式不下载 Three.js。

## 数据采集与发布门禁

Android 请求使用 `DS=Android`，iOS 请求使用 `DS=iOS`。采集器不再把“当前页少于 15 条”视为末页，而是严格解析并跟随 TapTap 返回的 `next_page` 游标。每条唯一游戏同时保存基于源游标和页内位置计算的榜位。

一次无上限采集必须满足以下合同：响应包含稳定的非负整数 `total`；游标正常结束；唯一游戏数等于 `total`；分页未超过安全上限。任何网络错误、响应结构错误、重复页、游标异常或数量不符都视为失败。主采集流程不再把失败榜单与旧缓存混合发布，避免“页面可打开但结论错误”。旧快照仍由 GitHub Pages 保留，只有完整的新快照才能替换它。

iOS 缺失数据曾污染私有变化状态，因此内部状态格式从 `1.0` 提升为 `1.1`。旧缓存会被当作不兼容状态；部署后的第一次完整采集只建立新基线，不生成补回游戏的虚假重新进榜或排名变化。公开变化 feed 仍为 `1.0`，前端无需迁移。

## 非新非热榜定义

计算分成两个集合：

1. 全局排除集：Android 与 iOS 所有热门榜、新品榜出现过的游戏 ID。
2. 候选集：当前展示范围内至少进入一个其他榜单的游戏 ID。

最终结果为候选集减全局排除集。即使用户只筛选 iOS，只要某游戏出现在 Android 热门榜或新品榜，也不能进入该榜。Python 产物和浏览器端筛选后的重算使用同一规则，并分别由单元测试覆盖。

## 游戏地图信息设计

地图采用与站点一致的浅色、扁平、细边框设计。横轴是对数热度，从低到高；纵向是有数据的类型泳道。游戏以真实图标显示，标记左下直接显示评分，右上显示榜单覆盖数。精确热度、日均热度、近 24 小时变化、平台覆盖和外部链接放在选中详情中。

默认视图每条泳道从低、中、高热度区间抽取代表游戏，避免把几十个标记堆在一起；右侧索引始终保留全部匹配游戏。搜索或选择类型后展开更多代表项，只保留有结果的泳道，并重新利用垂直空间。从索引选择未抽样游戏时，该游戏替换同泳道中热度位置最接近的代表项，标记总量不会突然增加。

Three.js 只在支持 WebGL、未启用省流量且设备能力足够时懒加载。它使用正交相机和静态实例网格在 DOM 标记下绘制低透明度的类型热度密度，不处理文字、点击或动画。DOM 始终拥有可访问名称、筛选、选中与详情。轻量模式保留全部功能，仅省略增强底层。

## 验证

自动验证包含：采集分页、设备标识、源榜位、总数合同、失败阻断、跨平台非新非热规则、内部基线迁移、地图抽样与选中替换、轻量模式不下载引擎、桌面和手机无横向溢出。发布前还需用真实 TapTap 接口逐一核对双平台 28 张榜单的 `collected == total`，并在浏览器检查地图筛选、索引选择、详情、后退状态及控制台错误。
