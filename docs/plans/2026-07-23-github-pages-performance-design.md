# GitHub Pages 首屏性能设计

## 目标

TTMRank 的主要瓶颈不是压缩后的字节数，而是 GitHub Pages/Fastly 在部分网络环境下每个独立请求都有数秒甚至更长的冷连接延迟。核心页面需要减少首次渲染前的请求数量，同时保留自动更新、完整原始排行榜和全站按需分析。

## 方案选择

采用“单文档首屏 + 内容哈希按需制品”的混合方案。部署构建器把页面首屏 CSS、已打包应用脚本和当前首屏数据内联到 HTML；浏览器收到 HTML 后即可渲染，不再等待 manifest 和第二份 JSON。全站分析、非默认排行榜和增强型 WebGL 引擎保持按需加载。

不采用服务工作线程，因为它只改善二次访问，并会引入旧数据和更新一致性风险。不把全站数据全部塞进 HTML，因为这会放大每次访问体积并破坏按需加载。

## 数据流

Actions 抓取完成后先生成兼容的 current JSON，再创建内容哈希副本和页面启动数据。分析页内联 manifest、TapTap制造分析和质量状态；首页与变化页内联 manifest 和变化情报；游戏地图内联 manifest 和制造范围视觉数据；原始排行内联 meta 以及 Android/iOS 热门榜。客户端优先读取 `#ttmrank-bootstrap`，缺失时仍使用原有静态 JSON 回退。

按需 JSON 使用哈希文件名，不依赖查询参数作为缓存键。网络读取设置超时并对瞬时错误重试一次，避免 CDN 连接永久挂起。

## 构建与安全

源码继续保存在 `app/`，构建输出写入临时目录供 Pages 上传。启动 JSON 会转义 `<`、`>`、`&` 和 Unicode 行分隔符，防止脚本标签提前结束。构建器校验 HTML 体积、哈希文件存在性和首屏启动数据完整性。

## 验证

单元测试覆盖启动数据零请求、哈希 URL 和瞬时失败重试；流水线测试覆盖部署前必须执行站点构建；Chromium 覆盖主要页面首屏无数据 JSON 请求、分析页切换全站、排行榜切换平台、390px 无横向溢出和控制台健康。
